# BABYMANIA ORGANIC SYSTEM — NIGHT EXECUTION PLAN
# תוכנית לילה מסודרת לייצוב ושיפור המערכת האורגנית

## CONTEXT (הקשר)

המטרה של המערכת:
לבנות מערך בלוגים ומאמרים לחנות BabyMania כדי לייצר:
- תנועה אורגנית (Organic Traffic / תנועה אורגנית)
- סמכות נושאית (Topical Authority / סמכות נושאית)
- קישורים פנימיים (Internal Linking / קישורים פנימיים)
- חיבור טבעי למוצרים (Product Bridges / גשרי מוצר)
- המרה לתנועה איכותית שמובילה לקנייה

המערכת כבר קיימת ועובדת חלקית.
כרגע לא בונים מערכת חדשה.
כרגע מייצבים, בודקים, מחזקים, ואז רק משפרים.

## CRITICAL RULES (חוקי חובה)

1. לא לבצע שינוי ארכיטקטורה רחב בלי הצדקה ברורה.
2. לא להוסיף 3 סוכנים חדשים יחד.
3. לעבוד לפי שלבים ברורים.
4. לא לנחש. לבדוק בפועל קבצים, חיבורים, פלטים, והרצות.
5. כל שלב חייב להסתיים בדוח מסודר.
6. כל שינוי חייב להיות מינימלי, מדויק, ועם סיבה.
7. אם משהו לא ברור — לסמן UNKNOWN ולא לנחש.
8. אסור לדלג על בדיקות.
9. לא לגעת במערכת דפי המוצר. עובדים רק על המערכת האורגנית.
10. אם נמצא כשל מערכת — מתקנים את ה־ENGINE (מנוע), לא את ה־OUTPUT (פלט) ידנית.

# PHASE 1 — BASELINE SYSTEM AUDIT
# שלב 1 — בדיקת עומק בסיסית של המערכת

## GOAL (מטרה)
לחשוף את מצב האמת של המערכת:
מה מחובר, מה לא מחובר, מה רץ, מה לא נאכף, ואיפה נשברת השרשרת.

## TASKS (משימות)
1. למפות את כל הסוכנים הפעילים בפועל.
2. למפות את כל קבצי הידע (Knowledge Files / קבצי ידע) והמחקרים.
3. לבדוק מי משתמש בכל קובץ ידע בפועל.
4. לבדוק אילו outputs (פלטים) נוצרים ואילו באמת נצרכים.
5. לבדוק האם יש סוכנים שהפלט שלהם נכתב אך לא משפיע על שום שלב.
6. לבדוק האם יש duplicate logic (לוגיקה כפולה) בין סוכנים.
7. לבדוק אילו קבצי publish / runner / scripts (סקריפטים) שולטים בפועל בפרסום.

## CHECKS (בדיקות)
- האם יש source of truth (מקור אמת) אחד לקבצי ידע?
- האם יש קבצי ידע כפולים או ישנים?
- האם יש סוכן שלא מחובר ל־pipeline (צינור עבודה)?
- האם יש סוכן שהפלט שלו נדרס או נעקף?
- האם יש פלטים שנוצרים אבל לא נצרכים?

## OUTPUT (פלט נדרש)
דוח בפורמט:
- SYSTEM STATE
- CONNECTION STATUS
- KNOWLEDGE USAGE
- FLOW BREAKS
- REDUNDANCY
- RISK LEVEL
- NEXT STEP

## DONE (הגדרת סיום)
השלב נגמר רק אם יש:
- מפת מערכת מלאה
- רשימת כשלי חיבור
- רשימת קבצי ידע לא מחוברים
- רשימת לוגיקות כפולות
- רשימת סיכונים אמיתיים

# PHASE 2 — ENFORCEMENT LAYER
# שלב 2 — שכבת אכיפה

## GOAL (מטרה)
להבטיח ששום מאמר לא מתפרסם אם הוא נכשל בבדיקות חובה.

## TASKS (משימות)
1. לאמת ש־Agent 10 (בודק קישורים / Link QA) מחובר כ־Hard Gate (שער חסימה קשיח) לפני כל publish.
2. לבדוק שכל publish script (סקריפט פרסום) באמת קורא את קובץ ה־QA לפני פרסום.
3. לבדוק שאין דרך לעקוף publish דרך script אחר או publish ידני.
4. לאתר publish paths (מסלולי פרסום) שלא מוגנים.

## CHECKS (בדיקות)
- האם publish נחסם אם אין QA file (קובץ QA)?
- האם publish נחסם אם QA != PASS?
- האם כל ה־HUB publish scripts (סקריפטי פרסום של אשכולות) מוגנים?
- האם יש מסלול publish לא מוגן?

## OUTPUT
- רשימת קבצי publish שנבדקו
- PASS / FAIL לכל קובץ
- אילו gaps (פערים) נשארו
- מה תוקן

## DONE
השלב נגמר רק אם:
- כל קבצי publish מחוברים לשער חסימה
- אין publish בלי PASS
- אין publish אם קובץ QA חסר

# PHASE 3 — VALIDATOR GAP AUDIT
# שלב 3 — בדיקת פערי הבודק

## GOAL
לזהות מה הבודקים בודקים ומה הם לא בודקים.

## TASKS
1. לבדוק את Agent 10 ברמת בדיקות טכניות.
2. לבדוק אילו בדיקות תוכן חסרות.
3. לסווג את החוסרים ל:
   - Technical Gaps (פערים טכניים)
   - Content Gaps (פערי תוכן)
   - Conversion Gaps (פערי המרה)

## CHECKS
בדוק האם המערכת מזהה:
- duplicate intent (כפילות כוונת חיפוש)
- image URL validity (תקינות קישורי תמונה)
- product handle validity לפני כתיבה
- עברית טבעית
- כותרות חזקות
- intro-box איכותי
- bridge טבעי למוצר
- חזרתיות בתוך מאמר
- חזרתיות בין מאמרים
- template fingerprint (טביעת תבנית חוזרת)

## OUTPUT
- VALIDATOR COVERAGE
- VALIDATOR GAPS
- PRIORITY GAPS (פערים בעדיפות גבוהה)

## DONE
השלב נגמר רק אם:
- יש רשימה ברורה של מה הבודק כן בודק
- יש רשימה ברורה של מה הבודק לא בודק
- יש חלוקה לפי עדיפות: קריטי / גבוה / בינוני

# PHASE 5 — CONTENT QA DECISION

- עברית לא טבעית

- intro-box חלש
- bridge חלש למוצר

- OPTION B: create Agent 10.5
- recommendation with reason

## DONE

# שלב 6 — תכנון מנגנון משימות

## GOAL

3. להגדיר status flow (זרימת סטטוסים):
   pending → running → done / failed / blocked

5. לא לממש מערכת מורכבת. רק skeleton (שלד) אמין.

## OUTPUT

- minimal runner plan
- failure handling
- stop conditions

## DONE

# EXECUTION RULES
# חוקי ביצוע

1. לעבוד שלב אחרי שלב.
2. לא לדלג קדימה.
3. לא לבצע תיקון בלי דוח מהשלב הקודם.
4. אם יש uncertainty (אי ודאות) — לעצור ולסמן UNKNOWN.
5. אם משהו כבר תוקן, לא לתקן שוב בלי סיבה.
6. כל שלב מחזיר דוח קצר ומובנה.
7. אם המשימה גדולה מדי ללילה אחד, לעצור אחרי סוף שלב ולהחזיר סטטוס ביניים מסודר.
